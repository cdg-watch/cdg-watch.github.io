# CDG Watch — エージェント向け指示書

コムデギャルソン(Comme des Garçons)関連情報を自動収集・整理する個人プロジェクト。
GitHub Pages で `index.html` が公開され、`data/items.json` を表示する。

> このファイルは監督エージェント(snowHSGG/snow-hq)が管理する。作業員は編集しないこと。
> `directive` ラベルの open Issue(または DIRECTIVES.md)は監督からの指示。毎回確認すること。

## 毎日のルーティン(日次エージェントの仕事)

1. `node scripts/collect.js` を実行して新着記事を取り込む。
   **収集が終わったら、要約に入る前にまず一度コミット・プッシュする**
   (`Daily update: 収集のみ` 等)。こうすれば以降の要約作業が途中でクラッシュ/時間切れに
   なっても、集めたデータは必ず保全される。**「必ずコミットまで到達する」ことが最優先。**
2. `data/items.json` の `summary: null` の項目について
   (**1回の実行で最大12件まで**。多すぎるとセッションが時間切れになりコミットに至らない。
   新しい記事から古い記事の順に処理し、直近の新着を優先、余った枠で過去のバックログを消化。
   残りは翌日以降に回す。重いと感じたら件数を減らしてよい):
   - 記事URLの内容を確認し、**自分の言葉で日本語1〜2文**の要約を `summary` に書く。
     記事本文のコピーは禁止(著作権対策)。事実の要約のみ。
   - `tags` を付ける。使用タグ: `新作`, `コラボ`, `ショー/コレクション`, `店舗`, `インタビュー`, `再販/セール`, `二次流通`, `その他`
   - **二次流通(リセール・中古・フリマの販売情報)は必ず `二次流通` タグを付ける**。
     公式ニュースと混ざらないようカテゴリを分離するのがこのサイトの方針
     (背景: ギャルソンのインラインは基本オンライン販売されないため、公式情報と
     二次流通情報は読者にとって別物)。collect.js が出典名から自動付与するが、
     漏れがあれば要約時に付け直すこと
   - CDG FREAK(cdg-freak.com)は店員が運営する信頼できる情報源。優先的に要約する
   - アクセスできない記事は `summary` に `"(要約取得不可)"` を入れてスキップ。
   - 記事ページを開いたついでに、`image` が null の項目は記事の og:image の **URL** を
     `image` フィールドに入れる(URLのみ。画像のダウンロード・リポジトリ保存は禁止)。
     ただし news.google.com のURLは中継ページでロゴ画像しか取れないため対象外。
     本文閲覧の結果、本物の記事URLが判明した場合は `url` を差し替えてよい。
3. 明らかな重複(同一内容の別ソース記事)があれば古い方を残して削除してよい。
   関連記事(同じコラボ/同じ発売の別ソース)には共通の `cluster` 文字列(例: `nike-airjordan11`)
   を各項目に付けてよい。サイトは将来これで束ね表示する想定。
4. **リリースカレンダーの更新**(`data/schedule.json`):
   - `productDrops`: その日要約した記事に発売日・入荷日が書かれていれば
     `{ "date": "YYYY-MM-DD", "title": "...", "brand": "...", "price": "...",
       "where": "...", "url": "..." }` を追加(未来の日付のみ。過去は入れない)。
   - `storeLaunches`: **月曜のみ**、CDG FREAK のスケジュールページ
     https://cdg-freak.com/comme-des-garcons-newarrival-schedule/ を読み、
     店頭の「立ち上がり日(新作が並ぶ日)」を
     `{ "date": "YYYY-MM-DD", "stores": ["青山店", ...], "note": "..." }` 形式で
     未来分だけ登録する(手書きテーブルなので内容を読んで正規化すること)。
   - **過去の項目は削除しない**(サイトの「過去の履歴」ビューで使う)。ただし1年より
     古いものは削除してよい。`updatedAt` を更新。日付は必ず YYYY-MM-DD。
5. **重要度スコア**: 各記事に `importance`(1〜3, 3が最重要)を付けてよい。新作の公式発表・
   大型コラボ・ショーは高め、小ネタは低め。サイトの「今日の要点」で優先表示される。
6. **日次ダイジェスト**: その日の要点を3〜5件、`data/digest-latest.md` に日本語で
   書き出す(見出し+1行要約+リンク)。フェーズ2のSNS投稿の下書きにも使う。
7. `directive` ラベルの open Issue(または DIRECTIVES.md の未対応項目)があれば読み、
   その日のうちに対応できる小さなものは1件対応する。データ作業は main 直接、
   コードの変更は必ずPR。対応したらIssueにコメントを残す
8. 変更を main に直接コミット・プッシュする(push 前に `git pull --rebase`)。
   コミットメッセージ: `Daily update: 新着N件`。
   **時間切れ対策**: 手順4〜7(カレンダー/重要度/ダイジェスト/directive)が重い場合は、
   要約とダイジェストを優先し、残りは翌日に回してでも必ずコミットして終えること。
   1回の実行で全部やり切るより、毎回確実にコミットが残る方が重要。

## 週次でやってよいこと(月曜のみ)

- 情報源の追加提案・サイト改善(UI、フィルタ、ソース追加など)を **PR として** 出す。
  main 直接プッシュはデータ更新のみ。コードの変更は必ずPR。

## 制約

- 依存パッケージは増やさない(現状ゼロ依存)。増やす場合はPRで理由を説明。
- 画像はリポジトリに保存しない(著作権対策。リンクのみ)。
  例外: `brand/` は自作のロゴ・SNS用素材置き場なので保存してよい。
- `data/items.json` は最大1000件。`scripts/collect.js` が自動で切り詰める。
